# TASK-028: Implement Semantic Versioning & Automated Changelog (release-please)

## Product Context
- Read: `docs/01-architecture/01-system-design-and-architecture.md` (Section: Engineering Maturity)
- Read: `docs/03-meta/agent-protocol.md` (Section: Commit Messages)

## Objective
Automate the release management process using Google's `release-please` GitHub Action. This enforces Conventional Commits, automatically bumps the Semantic Version (SemVer), generates a detailed `CHANGELOG.md`, and creates GitHub Releases, proving senior-level DevOps and release management maturity.

## Execution Boundaries
- You may ONLY create or modify files inside `.github/workflows/`, `.github/release-please-config.json`, `CHANGELOG.md` (generated by the action), and root `package.json` (for version tracking if needed).
- DO NOT modify Java source code, React components, or business logic.

## Context Files to Inject
- `docs/03-meta/agent-protocol.md`
- `.github/workflows/ci.yml`

## Acceptance Criteria

### 1. Conventional Commits Enforcement
1. Create a `.github/workflows/conventional-commit-lint.yml` workflow.
2. Use `amannn/action-semantic-pull-request@v5` to validate that all Pull Request titles follow the Conventional Commits standard (e.g., `feat:`, `fix:`, `chore:`, `docs:`).
3. Fail the check if the title does not match the standard.

### 2. Release Please Configuration
1. Create `.github/release-please-config.json`.
2. Configure it to recognize standard Conventional Commit types:
   - `feat` -> minor release
   - `fix` -> patch release
   - `BREAKING CHANGE` / `feat!` -> major release
   - `chore` -> exclude from changelog
3. Configure a manifest file to track the version (e.g., `version.txt` or root `package.json`).

### 3. Release Please GitHub Action
1. Create `.github/workflows/release-please.yml`.
2. Trigger on `push` to `main`.
3. Use `google-github-actions/release-please-action@v4`.
4. Configure it to run the release-please manifest command.
5. When a release is created, it should output the tag name and generate a draft GitHub Release with auto-generated release notes.
6. (Optional but recommended) Add a subsequent job in the workflow that triggers the CD pipeline (Task-024) only when a new release tag is published, separating feature pushes from production deployments.

### 4. Verification (Output Instructions)
- Document how a developer should structure their commit messages to trigger a patch vs. minor vs. major release.
- Confirm that the `CHANGELOG.md` will be automatically maintained at the repository root.

## Target File Paths
- `.github/workflows/conventional-commit-lint.yml`
- `.github/workflows/release-please.yml`
- `.github/release-please-config.json`
